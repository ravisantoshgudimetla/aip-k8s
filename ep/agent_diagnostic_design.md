# Design: AgentDiagnostic CRD

## Problem

Kubernetes-native AIP agents are stateless by default — controllers are reconciliation loops that hold no memory between runs. An agent that observes and diagnoses infrastructure state before deciding whether to act needs to persist that diagnostic state as a CRD.

Today, an agent author faces a choice that should not exist:

1. **Build a custom Diagnostic CRD** — the agent maintains two disconnected CRD schemas (its own `Diagnostic` and AIP's `AgentRequest`), two RBAC configurations, and two audit trails with no standard link between them.
2. **Misuse `AgentRequest`** — create an `AgentRequest` with no intended action, pulling the control plane into a decision it has no business making and bloating the governance record with non-governance data.

Every AIP adopter building a stateless diagnostic agent will hit this. Without a standard, every team builds their own variant. The schemas fragment, the audit trail breaks, and the "why did the agent act" question becomes harder to answer, not easier.

## Why Not Extend AgentRequest

`AgentRequest` is the wrong construct for observations. The reasons are mechanical, not aesthetic:

- `AgentRequest` requires control plane evaluation (SafetyPolicy, OpsLock). Observations have no blast radius — there is nothing for the control plane to evaluate or approve.
- `action` and `target` are required fields. An observation has no action and may not have a single target.
- Every `AgentRequest` generates `AuditRecord` entries. Injecting unbounded diagnostic volume into the governance audit trail pollutes the record auditors and SIEM tools depend on.
- The `AgentRequest` lifecycle state machine (`Pending → Approved → Completed`) has no meaningful interpretation for a read-only diagnostic.

The right construct for this is something the agent writes directly, with no controller involved — the same pattern Kubernetes uses for `v1.Event` and `coordination.k8s.io/v1.Lease`.

## Proposed Solution: `AgentDiagnostic` CRD

A new Kind in the `governance.aip.io/v1alpha1` API group. Agent-written directly. No controller. Immutable after creation.

### Design Decisions

**No controller.** The Kubernetes API server validates the schema and enforces RBAC. The agent writes the record once and it is done. The aip-k8s controller never reconciles `AgentDiagnostic` objects.

**Immutable after creation — spec and correlation labels.** Enforced via CEL validation rules. `spec` is fully immutable. The `aip.io/correlationID` label is separately locked because it is the primary evidence link between `AgentDiagnostic` and `AgentRequest` — allowing it to be silently rewritten after creation would allow decoupling of evidence without touching `spec`. Both rules are required; `self == oldSelf` alone only covers `spec`.

**`metadata.creationTimestamp` is the authoritative timestamp.** Set by the API server on write, cannot be faked by the agent.

**Links to `AgentRequest` via label, not field update.** The agent sets a shared `aip.io/correlationID` label on both the `AgentDiagnostic` and the subsequent `AgentRequest`. No back-link field on `AgentDiagnostic` is required — a single label selector retrieves the full chain.

**`details` is open JSON.** Different agent types produce different diagnostic data. The schema enforces structure around identity, type, and summary — but `details` is extensible, following the same model as `AgentRequest.spec.parameters`.

**Not visible to SafetyPolicy CEL expressions.** Policies evaluate `AgentRequest.spec` only. `AgentDiagnostic` records are agent-attested and informational. Allowing policies to join against agent-authored diagnostic data would give agent-authored content governance authority — collapsing the trust boundary the control plane exists to enforce.

### Schema

```go
// AgentDiagnosticSpec defines what the agent observed.
type AgentDiagnosticSpec struct {
    // AgentIdentity is the identity of the agent that recorded this diagnostic.
    // Must match the agentIdentity the agent uses in subsequent AgentRequests.
    AgentIdentity string `json:"agentIdentity"`

    // DiagnosticType categorizes the observation.
    // Recommended values: observation, diagnosis, signal, escalation.
    // Open string — adopters may define domain-specific types.
    DiagnosticType string `json:"diagnosticType"`

    // CorrelationID is a UUID generated by the agent before acting.
    // Set as a label (aip.io/correlationID) on this record and on any
    // subsequent AgentRequest to enable cross-resource querying.
    CorrelationID string `json:"correlationID"`

    // Summary is a human-readable description of what the agent observed.
    // Required. Consumed by dashboards and audit tooling.
    // +kubebuilder:validation:MaxLength=1024
    Summary string `json:"summary"`

    // Details contains agent-specific diagnostic data in open JSON.
    // Optional. Examples: confidence scores, evidence sources, alternatives considered.
    // +optional
    Details *apiextensionsv1.JSON `json:"details,omitempty"`
}

// AgentDiagnostic is the Schema for agent-originated diagnostic records.
// +kubebuilder:object:root=true
// +kubebuilder:resource:scope=Namespaced
// +kubebuilder:printcolumn:name="Agent",type=string,JSONPath=`.spec.agentIdentity`
// +kubebuilder:printcolumn:name="Type",type=string,JSONPath=`.spec.diagnosticType`
// +kubebuilder:printcolumn:name="Summary",type=string,JSONPath=`.spec.summary`
// +kubebuilder:printcolumn:name="Age",type=date,JSONPath=`.metadata.creationTimestamp`
type AgentDiagnostic struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`
    Spec AgentDiagnosticSpec `json:"spec,omitempty"`
}
```

### Immutability (CRD validation rules)

```yaml
x-kubernetes-validations:
  # Full spec immutability
  - rule: self.spec == oldSelf.spec
    message: "AgentDiagnostic spec is immutable after creation"
  # Correlation label immutability — must be locked separately;
  # self == oldSelf does not cover metadata.labels
  - rule: >
      !has(oldSelf.metadata.labels) ||
      !("aip.io/correlationID" in oldSelf.metadata.labels) ||
      oldSelf.metadata.labels["aip.io/correlationID"] == self.metadata.labels["aip.io/correlationID"]
    message: "aip.io/correlationID label is immutable after creation"
  # spec.correlationID must match aip.io/correlationID label at creation
  - rule: >
      !has(self.metadata.labels) ||
      !("aip.io/correlationID" in self.metadata.labels) ||
      self.metadata.labels["aip.io/correlationID"] == self.spec.correlationID
    message: "aip.io/correlationID label must match spec.correlationID"
```

### Standard Labels

| Label | Value | Set by |
|---|---|---|
| `aip.io/correlationID` | UUID generated by agent | Agent |
| `aip.io/agentIdentity` | Agent identity string | Agent |
| `aip.io/diagnosticType` | Mirrors `spec.diagnosticType` | Agent |

### Example CR

```yaml
apiVersion: governance.aip.io/v1alpha1
kind: AgentDiagnostic
metadata:
  name: diag-abc123
  namespace: production
  labels:
    aip.io/correlationID: abc123
    aip.io/agentIdentity: sre-diagnostic-agent-v1
    aip.io/diagnosticType: diagnosis
spec:
  agentIdentity: sre-diagnostic-agent-v1
  diagnosticType: diagnosis
  correlationID: abc123
  summary: "OOMKilled detected on payment-api (3 times in 10 min). Recommending restart."
  details:
    rootCause: OOMKilled
    occurrences: 3
    windowMinutes: 10
    confidence: 0.91
    alternativesConsidered:
      - action: restart
        selected: true
      - action: scale-out
        rejected: "memory leak affects all replicas"
    evidenceSources:
      - type: metrics
        ref: "prometheus://production/container_oom_events"
```

The agent then submits an `AgentRequest` carrying the same `aip.io/correlationID: abc123` label. The full incident chain is queryable with:

```bash
kubectl get agentdiagnostics,agentrequests,auditrecords \
  -n production \
  -l aip.io/correlationID=abc123 \
  --sort-by=.metadata.creationTimestamp
```

## Trust Model

| Resource | Author | Control Plane Involvement | Trust Level |
|---|---|---|---|
| `AgentDiagnostic` | Agent | None | Informational — agent-attested |
| `AgentRequest` spec | Agent | Evaluates, approves/denies | Informational — agent-attested |
| `AgentRequest` status | Control plane | Authoritative | Authoritative — governance decision |
| `AuditRecord` | Control plane | Authoritative | Authoritative — tamper-evident |

## Retention and Garbage Collection

Unlike `v1.Event` (which has a server-side TTL of ~1 hour), `AgentDiagnostic` records are part of the incident audit trail and must not disappear before the `AuditRecord` and `AgentRequest` they correlate with. Retention follows the same default as `AuditRecord`: **365 days minimum**, operator-configurable. The aip-k8s controller SHOULD implement a background GC loop that deletes `AgentDiagnostic` records past the configured retention window, consistent with how terminal `AgentRequest` and `AuditRecord` objects are purged.

## Relationship to the AIP Specification

`AgentDiagnostic` is a **Kubernetes-specific extension** of the aip-k8s implementation. It is not currently defined in the platform-agnostic AIP specification (`agent-intent-protocol` repo) because:

- The need for a persistence CRD is specific to stateless k8s controller-based agents. Non-k8s AIP implementations (REST, gRPC) can use databases or in-memory stores for agent-internal state.
- The spec governs what the control plane validates. `AgentDiagnostic` has no control plane involvement — it is outside the governance boundary by design.

If adoption patterns across multiple AIP implementations show a common need for standardized agent-originated diagnostic records, this may be promoted to the spec in a future version.
